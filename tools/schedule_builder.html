— step-by-step registration

1
Report Document
Which report should be generated?
The Document Name is the stable name of the report in the system. It is used to look up the document ID automatically at runtime — no IDs to manage manually.
2
Schedule Timing
When should this run?
Ad-hoc schedules fire once on the next trigger run, then automatically disable themselves. No timing configuration needed.
3
Report Parameters
What values does the report need?
Parameters are optional. Leave empty for reports that execute with no inputs. You can also import them from a Request JSON in Step 1.

For each parameter, set its value. Date parameters support tokens (like {{PREV_MONTH_END}}) that resolve automatically at runtime — see the Token Reference panel on the right.
4
Delivery
How should the report be delivered?
Configure the combined / bulk delivery here — this is what gets used when fan-out is None, and also for the combined row when fan-out is Individual + Combined.

For per-entity delivery (different email address / folder per value), configure that in Step 5.
✉
Email
Send the report as an email attachment to one or more recipients.
📁
Folder Drop
Write the report file to a network or cloud folder location.
✦
Both
Send by email AND drop to a folder — simultaneously.
✉ Combined Email Recipient
Who receives the combined report? Can be a static address, resolved from a view or function, or a dynamic SQL query.
For per-entity email addresses when using fan-out, configure those in Step 5.
The Folder Path and File Name can each be static, resolved from a view or function, or driven by a SQL query.
When fan-out is active in Step 5, you can also configure per-entity folder resolution there — whatever you set here applies to the combined / bulk row.

5
Fan-out Configuration
Optional — one report per value, or one combined?
Fan-out is optional. Skip this step for a single combined report delivered to the address / folder configured in Step 4.

Enable fan-out when a parameter has multiple values and each value should produce its own separate report — each delivered to that value's own specific email address or folder.

Combined row — when using Individual + Combined, the combined row always uses the Step 4 delivery config. The fields here only apply to the individual per-entity rows.

Only one parameter can drive fan-out at a time.
📋
None (Combined)
One report, all values together. Delivered to static recipients or folder.
👤
Individual
One report per value. Each delivered to that value's own recipient/folder.
👥
Individual + Combined
Individual reports for each value AND one combined report to the static recipient.
✉ Email — resolve per value
For each fan-out value, how should we find the recipient's email address?
Example: value BRM001 → look up in schdl.vw_BRMEmail → returns john@example.com
📁 Folder — resolve per value
For each fan-out value, how should we find the destination folder path?
Example: value BRM001 → look up in schdl.vw_BRMFolder → returns \\server\reports\BRM001\
🏷 Entity Label optional — for filenames and email personalisation
The human-readable name for each fan-out value. Becomes {{DISPLAYNAME}} in filenames and email text.
Example: value BRM001 → Entity Label John Smith → filename Report_John Smith_2026-05.xlsx
📄 File Name Override optional — extension added automatically
Override the default filename for each report. Leave blank to use the API default.
Supported placeholders: {{DISPLAYNAME}}  {{REPORTNAME}}  any date token e.g. {{PREV_MONTH_END}}
